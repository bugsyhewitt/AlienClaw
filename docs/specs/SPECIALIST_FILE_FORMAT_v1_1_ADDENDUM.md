# Specialist File Format Addendum v1.1

Implementation notes for Packet 11's TypeScript Specialist class.
These notes clarify spec ambiguities and record decisions. They do NOT
modify SPECIALIST_SPEC.md (which is locked).

---

## HEARTBEAT.md — spec format retained (not JSONL)

Packet 11's initial design proposed making HEARTBEAT.md a JSONL append-only
event log. SPECIALIST_SPEC.md §HEARTBEAT.md defines HEARTBEAT.md as a markdown
status file with State/Last-updated/Progress/Recent-activity/Blockers sections
that BossBot polls by reading fields.

**Decision: Follow the locked spec.** HEARTBEAT.md is a markdown status file,
rewritten atomically on each `updateHeartbeat()` call. Activities accumulate
in the Recent activity section (up to 10 kept in memory, 3 shown in the file)
without old entries being lost within a single campaign run.

Event-log semantics (born, summon-issued, summon-result, finalized, erased)
are not currently written anywhere on disk. If event replay becomes necessary,
a separate EVENTS.jsonl file can be added without touching HEARTBEAT.md's
contract. That addition would require a new addendum.

---

## MEMORY.md — append vs rewrite semantics

The spec says MEMORY.md is "updated freely by the Specialist." Packet 11 implements:

- **appendMemory(content)**: appends raw markdown text to the file. Fast path.
- **rewriteMemorySection(title, newContent)**: finds the `## <title>` heading
  and replaces its content up to the next `## ` heading (or EOF). Appends a
  new section if the title is not found.
- **recordResult(...)**: appends a structured "## Summon N — <type>" entry.

Nothing in MEMORY.md is ever deleted before `erase()`.

---

## Workspace base directory

Default: `~/.alienclaw/specialists/<campaign_id>/`.

Configurable via `SpecialistOptions.specialistsBaseDir` for test isolation.
Production code does not set this; it defaults to the home-relative path.
The base directory is created with `mkdirSync(..., { recursive: true })`.
Individual campaign directories are created with mode `0700`.

---

## Atomic writes

All file writes (initial birth files, HEARTBEAT.md updates) use a sibling
temp-file + rename pattern:

```
write to <workspace>/.tmp-<uuid>
rename to <workspace>/<filename>
```

This is atomic within the same filesystem (which is guaranteed since tmp file
and target are in the same directory). `appendFileSync` is used for MEMORY.md
additions — these are not atomic but MEMORY.md is the Specialist's scratch pad
and occasional partial writes are acceptable (the spec notes it's "erased with
the campaign directory" anyway).

---

## birth() idempotency

`birth(brief)` checks whether the workspace directory exists before creating it.
If it exists, it returns without writing any files. This prevents double-creation
if `runCampaign` is called twice with the same campaign_id (which SPECIALIST_SPEC
says "MUST NOT happen" but the code defends against it anyway).

---

## erase() behavior

1. Calls `updateHeartbeat('COMPLETE', 'Erased', 'Erasing workspace')` (best-effort)
2. Calls `rmSync(workspaceDir, { recursive: true, force: true })` — deletes the entire campaign directory
3. Sets `_erased = true`

The `~/.alienclaw/reports/<campaign_id>.md` campaign report is NOT deleted.
It belongs to BossBot and must survive erase per the spec.

The `erase()` method is idempotent: calling it twice does not throw.

---

## Campaign ID format in Packet 11

The spec defines the format as `CAMP_<8-char-uppercase-alphanumeric>`.
Packet 11's CreatorBot uses the existing `campaign_id` from `CampaignRequestMessage`,
which is generated by `newCorrelationId()` (a UUID). The CAMP_ prefix format
is adopted when the full multi-Martian campaign loop is implemented (future packet).
